# 02-强盛科技业务背景

## 一、企业概况

**强盛科技**是一家贸易型企业，2014-2015 年处于渠道扩张期，从华强北单档口交易进入"前店后仓 + 区域代理分销"阶段。

| 维度 | 情况 |
|---|---|
| 公司性质 | 贸易商行，开始正规化运营 |
| 经营模式 | 区域总代 + 前店后仓批发 |
| 门店形态 | 华强北展示档口 + 约 60 平米民房仓库 |
| 下游网络 | 30+ 地级市代理商 |
| 上游网络 | 50+ 固定供应商 |
| 商品规模 | 活跃 SKU 约 1500+ |
| 团队规模 | 10-15 人 |
| 业务线 | 渠道批发、档口零售、早期电商代发 |

## 二、组织分工

```
林强（老板）
├── 业务部：接代理商订单、报价、维护客户
├── 内勤/跟单：录单、协调、查询、跟进
├── 仓储部：拣货、打包、发货、盘点
└── 财务：收付款、核销、往来管理
```

## 三、当前业务痛点（五大类）

### 开单侧
- 录单前要靠人问价格、问库存、问客户欠款——没有统一数据
- 同一客户不同人容易报出不同价格
- 订单录进来后状态不透明，不知道谁在跟、跟到哪了

### 发货侧
- 销售认为"有货"，仓库实际不一定能发——库存口径不一致
- 发货动作和库存变化不是强绑定
- 短少、坏货、漏发后没有标准处理流程

### 收银侧
- 零售交易要求快，不能走批发那种重流程
- 议价和抹零是常态，但权限没有清晰边界
- 零售和批发若不共用库存，账面会不一致

### 入库侧
- 到货后靠人工点数和录单，容易出错
- 采购入库和应付形成容易脱节
- 采购数量、实收数量、入库数量容易混在一起——三个数量口径应分开
- 少收、拒收、部分到货没有稳定处理口径
- 采购价格变动时缺少统一记录和留痕

### 财务侧
- 应收应付看不清——不知道客户欠多少、欠了多久
- 回款登记和历史欠款核销效率低
- 财务只是被动收尾，没有进入业务闭环
- 私人转账认款困难，一笔款可能对应多笔单

## 四、业务场景频次判断

强盛科技当时最常发生的业务动作及重要性：

| 场景 | 频次 | 为什么重要 |
|---|---|---|
| 代理商接单 | 很高 | 直接影响主营收入，占日单量 80% |
| 批发开单 | 很高 | 是所有后续动作的起点 |
| 仓库出货 | 很高 | 直接影响履约和库存准确性 |
| 门店零售收银 | 高 | 现场效率要求高，走重流程会丢客 |
| 采购到货入库 | 中高 | 决定库存能否回补，影响发货能力 |
| 客户收款登记 | 中高 | 影响现金流和后续放货 |
| 门店与后仓调拨 | 中 | 支撑前店后仓协同 |
| 销售/采购退货 | 中/中低 | 处理售后和不良品 |
| 盘点 | 低到中 | 校正账实差异 |
| 报损 | 低到中 | 记录损耗和异常 |

## 五、一期项目判断

在资源有限、基建未成熟的前提下，一期最忌讳：
- 还没把基础资料和基础单据立起来就想做复杂规则
- 还没把高频主链路跑通就先上高级分析
- 还没让销售、仓库、财务进入同一套底账就开始谈智能补货

**一期目标：先做一个最小可运行的基础进销存闭环。** 优先解决：订单能不能顺利录进去 → 库存能不能随着业务动作变化 → 仓库能不能按单发货 → 财务能不能看清基本往来。

**一期先做：**
- 商品、客户、供应商、仓库基础资料
- 批发销售订单 + 销售出库
- 零售收银
- 采购订单 + 采购入库 + 采购退货
- 基础库存查询 + 调拨 + 盘点 + 报损
- 收款付款登记 + 基础应收应付查询

**一期不做：**
- 自动补货建议、安全库存预警（依赖数据积累，一期数据不够）
- 信用额度自动硬拦截（需要应收链路先稳定）
- 复杂审批流（一期先做提醒和基础权限）
- 复杂经营分析报表（先让数据进系统，再谈分析）
- 电商代发扩展（当时不是主营盘）
