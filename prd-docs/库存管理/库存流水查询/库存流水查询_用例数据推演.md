## 3. 推演用例

---

### 案例 1：多条件交叉筛选（商品 + 仓库 + 变动方向）

> **目标**：验证当用户同时选择多个筛选字段时，系统执行 `AND` 逻辑进行过滤，返回数据子集正确。

**场景**：
仓管员在页面选择：
- 仓库 = `WH001 民房一号仓`
- 商品 = `SKU001 华强北特种接插件`
- 变动方向 = `IN=入库（+）`
- 时间范围 = 2026-07-01 至 2026-07-04（共 4 天，在365天跨度内）

**推演步骤**：

| 步骤 | 执行方 | 操作 | 变动数据结果 (行数: 5 $\rightarrow$ N) | 验证点 |
| :--- | :--- | :--- | :--- | :--- |
| 1 | 用户 | 输入并选择条件，点击搜索 | 系统接收过滤字段组合：<br>`whse = 'WH001'`<br>`AND SKU = 'SKU001'`<br>`AND qty > 0` | 校验时间跨度天数，通过。 |
| 2 | 系统后台 | 执行 SQL 查询并返回 | 列表中仅返回符合上述所有条件的以下 2 行：<br>1. **FL20260704-00000001** (2026-07-04, 变动数量 = +6)<br>2. **FL20260701-00000002** (2026-07-01, 变动数量 = +20) | 过滤精确性：<br>- FL-005 (SKU001，WH002) 因仓库不符被排查。<br>- FL-012 (SKU001，WH001，-5) 因变动方向为出库被排查。<br>- FL-011 (SKU002，WH001) 因商品不符被排查。 |
| 3 | 前端界面 | 渲染表格列，更新底部汇总 | 页面显示 2 条记录，变动数量以带 `+` 绿色字体展现 | 发生时间由新到旧排列正确。 |

**过滤结果校验**：

*   **预期列表行数**：2 行
*   **命中流水号清单**：`FL20260704-00000001`、`FL20260701-00000002`

### 案例 2：单据类型与模糊单号检索

> **目标**：验证在查询具体某类型业务时，配合单号模糊匹配能够准确定位。

**场景**：
财务对账，选择：
- 单据类型 = `PI=采购入库`
- 来源单号 = `202607` (要求模糊查询 2026年7月的入库单流水)

**推演步骤**：

| 步骤 | 执行方 | 操作 | 变动数据结果 | 验证点 |
| :--- | :--- | :--- | :--- | :--- |
| 1 | 用户 | 选择单据类型，输入 `202607`，点击搜索 | SQL 逻辑中拼接：<br>`type = '采购入库' AND doc_no LIKE '%202607%'` | 模糊匹配前缀支持。 |
| 2 | 系统后台 | 检索底表并返回集 | 列表中返回符合条件的三条流水：<br>1. **FL20260704-00000001** (来源: PI20260704-0001)<br>2. **FL20260702-00000011** (来源: PI20260702-0003)<br>3. **FL20260701-00000002** (来源: PI20260701-0001) | 过滤精确性：<br>- FL-012 (来源 SOO...) 因类型不符被排除。<br>- FL-005 (来源 RS...) 因类型不符被排除。 |

**过滤结果校验**：
*   **预期列表行数**：3 行
*   **命中流水号清单**：`FL20260704-00000001`、`FL20260702-00000011`、`FL20260701-00000002`。

## 4. 设计走查清单

*   [x] **日期超期强拦截**：选择跨度大于 365 天时，是否成功进行红字报错并置灰查询与导出按钮。
*   [x] **出入库数量方向**：入库流水数量全部正确带 `+` 符号，出库流水全部正确带 `-` 符号，无合并混淆。
*   [x] **变动类型正确映射**：底表中的 `RS` 前缀在列表页上是否正确汉化渲染为“零售单成交”。
*   [x] **下钻跳转路由正确**：点击 `PI20260704-0001` 时，是否路由下钻跳转至采购入库单详情页。
*   [x] **无数据导出控制**：在案例4（无匹配结果）的场景下，表格展示空状态，工具条中的“导出”按钮是否被置为 Disabled 禁用状态。
