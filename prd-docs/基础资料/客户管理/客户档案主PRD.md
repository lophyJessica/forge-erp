# 客户档案主PRD

> **版本**：V1.1 | 2026-04-19
> **读者**：产品经理、前端工程师、测试工程师
> **说明**：客户档案属于主数据 CRUD，当前补充模块级主PRD，用于统一客户档案的业务边界、共性规则和验收重点。

---

## 1. 业务背景

强盛科技的渠道批发业务本质上是"卖给不同等级客户，并按不同价格、账期和信用方式承接订单"。  
如果客户资料不统一，就会直接出现：

- 同客户不同人报出不同价格
- 客户账期和信用额度不透明
- 收货信息分散，发货容易出错

因此，客户档案的目标是先统一"卖给谁、按什么价格卖、什么条件下允许挂账"。

---

## 2. 功能范围

**In Scope**

- 客户基础资料维护
- 客户等级、价格级别、结算方式、账期和信用额度维护
- 默认收货信息维护
- 启用 / 停用管理

**Out of Scope**

- 客户分层审批流程
- 自动信用强控和特批流
- 客户画像和复杂标签体系
- 往来摘要和经营分析看板

## 5. 核心业务规则

| 规则ID | 规则 |
| :--- | :--- |
| R01 | 客户编码必须唯一，且一旦被销售单据引用后不可修改。 |
| R02 | 客户必须维护客户等级、价格级别、结算方式。账期天数和信用额度仅在"账期结算"时必填；现结客户对应字段固定为 `0`。 |
| R03 | 现结客户的账期天数固定为 `0`，信用额度默认 `0.00`，不可手动编辑；切换为现结时系统自动重置。 |
| R04 | 销售订单引用客户时，需要固化客户名称、联系人、联系电话、收货信息、价格级别、结算方式、账期天数等快照。 |
| R05 | 停用客户不可在新建销售订单和收款登记中继续选择。 |

## 6. 状态与动作规则

| 状态 | 含义 |
| :--- | :--- |
| 启用 | 可在新业务中继续选择 |
| 停用 | 不可在新业务中继续选择 |

| 动作 | 启用 | 停用 | 说明 |
| :--- | :---: | :---: | :--- |
| 查看 | ✅ | ✅ | 始终允许 |
| 编辑 | ✅ | ✅ | 编码字段受引用约束 |
| 停用 | ✅ | ❌ | 停用时需填写原因 |
| 启用 | ❌ | ✅ | 支持恢复 |
| 删除 | ❌ | ❌ | 主数据不提供删除能力。若对象不再使用，统一通过"停用"处理；已被业务单据引用的数据必须长期保留，确保历史记录可追溯。 |

## 7. 动作级规则

### 7.1 新增规则

| 规则ID | 规则 |
| :--- | :--- |
| A01 | 新增客户时必须填写：客户编码、客户名称、客户等级、联系人、联系电话、价格级别、结算方式。 |
| A02 | 客户编码新增时必须做唯一性校验，重复则阻断保存。 |
| A03 | 收货信息可为空，若未维护则后续在销售订单中补充。 |
| A04 | 新增时客户默认为"启用"状态，不允许在新增表单中直接选择状态。 |

### 7.2 编辑规则

| 规则ID | 规则 |
| :--- | :--- |
| A11 | 已被销售单据引用的客户，编码不可编辑。 |
| A12 | 客户名称、等级、联系人、电话、收货信息、价格级别、结算方式、账期、信用额度允许编辑。 |
| A13 | 已停用客户仍允许查看和编辑非编码类字段。 |

### 7.3 停用 / 启用规则

| 规则ID | 规则 |
| :--- | :--- |
| A21 | 停用客户时必须填写停用原因。 |
| A22 | 停用后不可在新建销售订单和收款登记中继续选择。 |
| A23 | 已停用客户允许恢复启用，恢复后可重新在新单中选择。 |
| A24 | **启停用操作通过独立的"停用/启用"动作按钮触发，不能在编辑表单中直接修改状态字段。** 停用时系统弹出确认弹窗，要求用户填写停用原因后方可完成操作。 |

## 8. 下游引用影响规则

| 引用模块 | 引用方式 | 说明 |
| :--- | :--- | :--- |
| 销售订单 | 选择客户 | 引用客户编码、名称、联系人、联系电话、收货信息、价格级别、结算方式、账期快照 |
| 收款登记 | 选择客户 | 引用客户编码、名称、结算方式 |
| 客户往来查询 | 作为往来主体 | 引用客户编码、名称 |

补充规则：

- 已被业务单据引用的客户，历史单据继续展示原客户信息
- 停用客户不影响历史订单和往来查询展示
- 当前一期不在客户档案中展开复杂客户画像、经营标签、自动风控能力

---

## 9. 异常与边界规则

| 场景 | 处理口径 |
| :--- | :--- |
| 客户编码重复 | 阻断保存，提示"客户编码已存在" |
| 停用时未填写原因 | 阻断停用，弹窗内提示补充停用原因 |
| 已引用客户尝试修改编码 | 阻断修改，提示该客户已被业务单据引用 |
| 结算方式改为现结 | 自动将账期天数重置为 `0`（只读展示），信用额度重置为 `0.00`（只读展示） |
| 客户名称重复但编码不同 | 当前一期允许，但建议人工确认是否重复建档 |
| 收货信息为空 | 允许保存；后续销售订单中补充即可 |

## 10. 权限设计

> **课程口径说明**：一期进销存课程场景较小，业务员/内勤统一负责客户关系管理，故允许业务员新增和编辑客户档案。实际企业场景中，客户信用额度和价格级别通常需要销售主管或管理员审批，业务员仅有查看权限或有限编辑权限。本模块的权限设计为课程简化版，不作为真实系统设计参考。

| 操作 | 业务员/内勤 | 财务 | 管理员 |
| :--- | :---: | :---: | :---: |
| 查看 | ✅ | ✅ | ✅ |
| 新增 | ✅ | ✅ | ✅ |
| 编辑 | ✅ | ✅ | ✅ |
| 启用/停用 | ❌ | ❌ | ✅ |

## 11. 反馈与验收重点

### 11.1 关键反馈

| 操作 | 反馈方式 | 文案 |
| :--- | :--- | :--- |
| 保存成功 | Toast | 保存成功 |
| 启用成功 | Toast | 客户已启用 |
| 停用成功 | Toast | 客户已停用 |
| 编码重复 | 字段内红字提示 | 客户编码已存在 |
| 已引用不可改编码 | Toast | 该客户已被业务单据引用，编码不可修改 |
| 结算方式切换 | 联动更新 | 切换结算方式后同步刷新账期与信用字段 |
| 停用时原因未填 | 弹窗内字段红字提示 | 停用原因不能为空 |

### 11.2 验收重点

| # | 验收项 | 验收标准 |
| :--- | :--- | :--- |
| V01 | 编码唯一性 | 客户编码不允许重复 |
| V02 | 价格与信用规则完整 | 客户等级、价格级别、账期、信用额度字段齐全且必填逻辑正确 |
| V03 | 条件联动 | 现结和账期结算的字段联动正确；切换现结时账期和信用额度自动重置且置为只读 |
| V04 | 快照规则明确 | 销售订单引用客户后快照口径清晰 |
| V05 | 停用限制 | 停用客户不可继续用于新单 |
| V06 | 启停用操作路径 | 启停用通过操作按钮弹窗触发，不在编辑表单中出现状态字段 |
