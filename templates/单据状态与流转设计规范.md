# 通用状态机设计原则

## 一、目标

当前项目的状态设计，优先保证：

1. 状态能闭环
2. 状态和动作对应
3. 状态名容易理解
4. 不为训练营项目引入不必要复杂度

---

## 二、设计原则

### 2.1 一个状态必须回答三件事

- 现在处于什么阶段
- 允许做什么动作
- 不能做什么动作

### 2.2 先少后多

- 一期先用够用的状态
- 不为了显得专业而拆过细状态

### 2.3 状态和动作必须成对设计

不要只写状态名，不写动作约束。

### 2.4 状态变更必须通过动作触发，不允许在表单中直接修改

> 这是本项目全局约束，适用于所有模块。

- 状态字段**不能**作为表单下拉直接修改（如在编辑页放一个"状态"Select）
- 状态变更必须通过**独立的动作按钮**触发（如"停用"按钮、"审核"按钮）
- 动作触发后，系统弹出确认弹窗，弹窗内可包含附加必填字段（如停用原因）

**反例**（禁止）：

```
编辑页表单中有一个 [状态: 启用 ▼] 下拉，用户可直接改成"停用"再保存
```

**正例**（要求）：

```
详情页/列表操作列有"停用"按钮 → 点击弹出确认弹窗（含"停用原因"必填项）→ 确认后完成状态变更
```

### 2.5 主数据模块"删除"的统一处理方式

主数据（仓库、供应商、客户、商品等）**一律不提供删除能力**。理由：

- 已被业务单据引用的数据必须长期保留，确保历史记录可追溯
- 误删主数据会导致历史单据数据孤立

统一做法：若对象不再使用，通过"停用"操作处理。停用后：

- 不可在新建业务单据中选择
- 历史单据仍保留完整数据
- 可随时恢复启用

---

## 三、主数据启停用状态机标准模式

> 适用范围：仓库档案、供应商档案、客户档案、商品档案（以及后续新增的主数据模块）。

### 状态定义

| 状态 | 枚举值 | 含义 |
| :--- | :--- | :--- |
| 启用 | `ENABLED` | 可在新业务单据中继续选择 |
| 停用 | `DISABLED` | 不可在新业务单据中继续选择 |

### 动作矩阵（标准模板）

| 动作 | 启用 | 停用 | 说明 |
| :--- | :---: | :---: | :--- |
| 查看 | ✅ | ✅ | 始终允许 |
| 编辑 | ✅ | ✅ | 编码字段受单据引用约束 |
| 停用 | ✅ | ❌ | 触发弹窗，弹窗内停用原因必填 |
| 启用 | ❌ | ✅ | 触发弹窗（纯确认，无附加字段） |
| 删除 | ❌ | ❌ | 主数据不提供删除，统一用"停用"替代 |

### 新增时默认状态规则

- 新增主数据时，默认状态为"启用"
- 新增表单中**不展示状态字段**，不允许用户在新增时选择状态

---

## 四、建议检查项

- 是否存在永远到不了的状态
- 是否存在状态之间回不来的断点
- 是否存在两个名字不同、含义其实一样的状态
- 是否存在页面上允许动作和状态口径冲突
- 是否存在通过表单直接修改状态字段的设计（违反原则 2.4）
- 新增表单是否展示了状态字段（违反新增默认规则）
