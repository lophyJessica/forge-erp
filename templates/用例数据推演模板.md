# 用例数据推演模板

> **定位**：方法论第⑤步的产出物——第二道验证关卡。用具体数据代入流程，验证「账算得平、状态闭得住、字段接得上」
>
> **生成方式**：由 AI 基于《字段清单》《主PRD》《流程推演》完整生成；人负责走查，发现漏洞后回写问题记录
>
> **在方法论中的位置**：中观层 → 第⑤步 → 验证关卡②
>
> **与流程推演的区别**：
> - **流程推演**（Step③）：验证业务场景能不能走通，不要求字段级精确
> - **用例数据推演**（Step⑤）：带着具体数值、金额、状态走一遍，验证数据闭环
>
> **关卡规则**：任何案例不通过 → 回到字段清单或主PRD修正 → 重新推演不通过的案例

---

## 使用说明

### AI 生成时必须覆盖的内容

1. **测试数据集**：主数据 + 配置数据，数值要真实合理（不用 `值1`、`值2`）
2. **推演案例**：至少覆盖以下 5 种类型（无关案例可删，但若删除需在模板里注明原因）：
   - 案例1：正常流程（Happy Path）
   - 案例2：部分完成（如部分收货、分批执行）
   - 案例3：异常中断（取消/撤回）
   - 案例4：逆向处理（红冲/反审核，如有）
   - 案例5：边界/并发（如有幂等、并发约束）
3. **每个案例**必须包含：推演步骤表 → 状态快照 → 数字校验（如涉及金额/数量计算）→ 走查清单 → 结论
4. **推演发现的问题**：AI 在推演过程中自行识别的逻辑漏洞，全部记录在§4

### 人走查时的职责

走查不是重新推演，而是**带着问题清单核查 AI 的输出**。每个案例末尾的「走查清单」就是人的核查入口：

- ✅ 方框全打勾 → 案例通过
- ❌ 有任何一项未通过 → 在§4记录问题，标注需修正的文档位置

### 推演发现问题后的回写映射

推演的目标不是只输出一份“问题清单”，而是把问题准确回写到对应文档。

| 发现的问题类型 | 优先回写文档 | 典型问题 |
| :--- | :--- | :--- |
| 字段定义不清 / 缺字段 / 枚举不全 | 字段清单 | 编码格式、枚举值、必填性、回写字段缺失 |
| 状态流转不清 / 关闭条件不清 / 规则冲突 | 主PRD | 状态机、动作矩阵、业务规则、异常处理口径 |
| 页面交互不清 / 校验时机不清 / 保存后行为不清 | 前端Demo PRD | 字段长度限制、特殊字符限制、保存后跳转、错误提示位置 |
| 纯实现问题 / 组件能力不足 / 缓存逻辑缺失 | 前端代码任务 | localStorage、Toast、只读态、TextArea行为 |

回写原则：

- **字段问题** 不回写到主PRD
- **规则问题** 不回写到字段清单
- **页面行为问题** 不回写到字段清单或主PRD
- **纯实现问题** 不在业务文档里伪造规则，应单独形成前端改造任务

## 数据变化写法约定

> AI 在填写「推演步骤」的「数据变化」列时，必须遵循以下格式，不得只写一句描述。

| 场景 | 标准写法 | ❌ 错误写法 |
| :--- | :--- | :--- |
| 字段新生成 | `单据号 = PO20260419-0001（新生成）` | `单据号生成` |
| 字段状态变化 | `单据状态：草稿 → 待审核` | `状态更新` |
| 数值累加 | `累计入库数量：0 → 8（回写）` | `数量更新` |
| 字段变化+公式 | `金额：100 × 8 = 800.00` | `金额自动计算` |
| 无变化 | `—` | 留空 |
| 多个字段同时变化 | 分行列出，每行一个字段 | 合并写在一行 |

### 案例 1：正常流程

> **目标**：验证主流程（Happy Path）的状态流转、金额计算、数量传递全部正确。

**场景**：{一句话描述，如「采购 SKU001×10、SKU002×5，全量收货并入库」}

**初始数据**：

| 字段 | 值 |
| :--- | :--- |
| 供应商 | S001 {供应商名称} |
| 仓库 | WH001 {仓库名称} |
| 业务日期 | {YYYY-MM-DD} |
| {业务类型} | {值} |
| 商品明细 | SKU001 × 10；SKU002 × 5 |

**推演步骤**：

| 步骤 | 操作人/系统 | 操作 | 数据变化（字段 = 旧值 → 新值） | 验证点 |
| :--- | :--- | :--- | :--- | :--- |
| 1 | 采购员 | 新增单据，填写基本信息 | `单据号 = PO20260419-0001（新生成）`<br>`单据状态：（无）→ 草稿` | 单据号格式符合《编号规则》 |
| 2 | 采购员 | 提交审核 | `单据状态：草稿 → 待审核` | 商品明细不为空；必填字段已填 |
| 3 | 主管 | 审核通过 | `单据状态：待审核 → 已审核`<br>`SKU001预占库存：{旧} → {旧+10}`<br>`SKU001可用库存：{旧} → {旧-10}` | 库存预占数量变化正确 |
| 4 | 系统 | 下推生成收货通知单 | `通知单号 = RN20260419-0001（新生成）`<br>`SKU001可下推数量：10 → 0`<br>`通知单 SKU001数量：继承 = 10` | 字段继承正确；可下推数量清零 |
| 5 | 仓库 | WMS回传入库结果 | `累计入库数量（SKU001）：0 → 10`<br>`累计入库数量（SKU002）：0 → 5`<br>`SKU001预占：{旧} → {旧-10}`<br>`SKU001账面库存：{旧} → {旧+10}` | 回写逻辑正确；账面库存变化正确 |
| 6 | 系统 | 自动判断关闭状态 | `SKU001行关闭状态：未关闭 → 已关闭`<br>`SKU002行关闭状态：未关闭 → 已关闭`<br>`整单关闭状态：未关闭 → 已关闭` | 所有行关闭后整单自动关闭 |

**金额校验**：

| SKU | 数量 | 含税单价 | 税率 | 不含税单价 | 金额（不含税） | 税额 | 价税合计 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| SKU001 | 10 | {X} | 13% | `{X} ÷ 1.13 = {不含税}` | `{不含税} × 10` | `价税合计 - 金额` | `{X} × 10` |
| SKU002 | 5 | {Y} | 13% | `{Y} ÷ 1.13 = {不含税}` | `{不含税} × 5` | `价税合计 - 金额` | `{Y} × 5` |
| **合计** | — | — | — | — | **{合计}** | **{合计}** | **{合计}** |

> **计算规则验证**：
> - 不含税单价 = 含税单价 ÷ (1 + 税率)，结果保留 6 位小数
> - 金额（不含税）= 不含税单价 × 数量，保留 2 位小数
> - 价税合计 = 含税单价 × 数量，保留 2 位小数
> - 税额 = 价税合计 - 金额（差额法，不对税额单独舍入）

**状态快照**（每步结束后的完整状态）：

| 步骤 | 单据状态 | 关闭状态 | SKU001累计入库 | SKU001可下推 |
| :--- | :--- | :--- | :--- | :--- |
| 步骤1（新增后） | 草稿 | 未关闭 | 0 | 10 |
| 步骤2（提交后） | 待审核 | 未关闭 | 0 | 10 |
| 步骤3（审核后） | 已审核 | 未关闭 | 0 | 10 |
| 步骤4（下推后） | 已审核 | 未关闭 | 0 | 0 |
| 步骤5（回写后） | 已审核 | 未关闭→已关闭 | 10 | 0 |
| 步骤6（系统判断后） | 已审核 | 已关闭 | 10 | 0 |

**走查清单**（人逐项核对）：

- [ ] 单据号格式与《编号规则》一致
- [ ] 金额表中每行的计算过程和结果正确（重点检查精度处理——6位保留 vs 2位保留）
- [ ] 税额是否使用差额法（= 价税合计 - 金额），而非独立计算再相加
- [ ] 下推时字段继承的字段列表完整（是否有遗漏的字段没继承）
- [ ] 回写后库存变化方向正确（预占减少，账面增加）
- [ ] 全量入库后，整单关闭状态自动变「已关闭」
- [ ] 状态快照表中没有中间状态跳跃（如从草稿直接变已审核）

**结论**：✅ 通过 / ❌ 不通过（{原因}）

### 案例 2：部分完成

> **目标**：验证「不是全部完成」的场景——可下推数量计算、行级关闭状态自动判断、多次回写累加。

**场景**：{如「SKU001订单10个，第一次入库8个，第二次入库2个；SKU002延迟交货」}

**初始数据**：同案例1

**推演步骤**：

| 步骤 | 操作人/系统 | 操作 | 数据变化 | 验证点 |
| :--- | :--- | :--- | :--- | :--- |
| 1 | WMS | 第一次回传：SKU001实收 8 | `累计入库数量（SKU001）：0 → 8`<br>`可下推数量（SKU001）：10 → 2` | 剩余可下推 = 订单数 - 累计入库数 |
| 2 | 采购员 | 再次下推，生成第二张通知单 | `通知单号 = RN20260419-0002（新生成）`<br>`SKU001数量 = 2` | 第二张通知单数量 = 剩余可下推数 |
| 3 | WMS | 第二次回传：SKU001实收 2 | `累计入库数量（SKU001）：8 → 10`<br>`可下推数量（SKU001）：2 → 0`<br>`SKU001行关闭状态：未关闭 → 已关闭` | 行级自动关闭：累计 = 订单数时触发 |
| 4 | 系统 | 判断整单关闭状态 | `整单关闭状态：未关闭`（SKU002仍未完成） | SKU002未关闭，整单不应关闭 |

**状态快照**：

| 步骤 | SKU001累计入库 | SKU001可下推 | SKU001行关闭 | SKU002行关闭 | 整单关闭 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 步骤1后 | 8 | 2 | 未关闭 | 未关闭 | 未关闭 |
| 步骤2后 | 8 | 0 | 未关闭 | 未关闭 | 未关闭 |
| 步骤3后 | 10 | 0 | **已关闭** | 未关闭 | 未关闭 |
| 步骤4后 | 10 | 0 | 已关闭 | 未关闭 | 未关闭（等SKU002） |

**数字校验**：

| 维度 | 预期值 | 计算依据 |
| :--- | :--- | :--- |
| SKU001 可下推数量（步骤1后） | 2 | 10 - 8 = 2 |
| SKU001 可下推数量（步骤3后） | 0 | 10 - 10 = 0 |
| 整单关闭状态（步骤4后） | 未关闭 | SKU002行未关闭，整单不触发 |

**走查清单**：

- [ ] 两次回写的累积入库数量是叠加而非覆盖（8 + 2 = 10，不是直接变 2）
- [ ] 可下推数量的计算公式：订单数 - 累计入库数（是否有其他减项？）
- [ ] SKU001 行关闭条件：累计入库 = 订单数时，行自动关闭（是系统自动还是需要手动触发？）
- [ ] 整单关闭状态：仅当所有行均已关闭时才触发（有一行未关闭则整单不关闭）
- [ ] 第二次下推时，通知单数量正确等于剩余可下推数

**结论**：✅ / ❌

### 案例 3：异常中断（取消/撤回）

> **目标**：验证取消或撤回后的数据一致性——库存是否释放？下游是否级联处理？金额和数量是否归位？

**场景**：{如「已审核，已下推一张通知单（待确认状态），采购员申请取消整单」}

**推演步骤**：

| 步骤 | 操作人/系统 | 操作 | 数据变化 | 验证点 |
| :--- | :--- | :--- | :--- | :--- |
| 1 | 系统 | 检查下游状态 | — | 通知单状态 = {待确认}，是否在可级联取消范围？ |
| 2 | 系统 | 级联取消通知单 | `通知单状态：待确认 → 已取消` | 取消前先处理下游，再取消本单 |
| 3 | 系统 | 取消本单 | `单据状态：已审核 → 已取消`<br>`单据关闭状态：未关闭 → 已关闭`<br>`SKU001预占库存：{旧} → {旧-10}`<br>`SKU001可用库存：{旧} → {旧+10}` | 取消后预占释放，可用库存恢复 |

**状态快照**：

| 时间点 | 本单状态 | 本单关闭状态 | 通知单状态 | SKU001预占 | SKU001可用 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 取消前 | 已审核 | 未关闭 | 待确认 | +10（预占中） | -10 |
| 取消后 | 已取消 | 已关闭 | 已取消 | 0（已释放） | 恢复 |

**边界验证**：

| 场景变体 | 预期结果 | 验证依据 |
| :--- | :--- | :--- |
| 通知单状态为「已确认（执行中）」时尝试取消 | **阻断**，提示「存在执行中的通知单，请先处理」 | PRD §取消规则 |
| 通知单已全部完成（已关闭）时尝试取消 | **阻断**，提示「单据已全量执行，不可取消，如需纠错请走红冲流程」 | PRD §取消规则 |

**走查清单**：

- [ ] 取消的前置检查：系统是否先检查下游状态再决定是否继续？（不能直接取消不处理下游）
- [ ] 库存预占：取消后预占是否释放？可用库存是否正确恢复？
- [ ] 关闭状态：取消后整单关闭状态是否同步变「已关闭」？
- [ ] 取消终态：已取消后是否还可以被编辑/提交？（应为不可操作的终态）
- [ ] 边界场景：下游不可取消时，系统是否给出清晰的阻断提示（含具体的下游单据类型和状态）？

**结论**：✅ / ❌

### 案例 4：逆向处理（红冲/反审核）

> **目标**：验证红冲或反审核后的数据回滚——从「已完成」倒回到可修正的状态，数据是否能对齐。
>
> **如模块不涉及逆向处理，删除本案例，并在此注明原因**（如：本模块无反审核功能，入库后的纠错通过「其他出库单」对冲）。

**场景**：{如「入库单已审核并影响库存，发现数量录入错误，需走红冲流程」}

**推演步骤**：

| 步骤 | 操作人/系统 | 操作 | 数据变化 | 验证点 |
| :--- | :--- | :--- | :--- | :--- |
| 1 | 主管 | 发起红冲，生成负数入库单 | `红冲单号 = PO20260419-0001-R（新生成）`<br>`红冲数量 = -{原数量}` | 红冲单号格式；数量取负值 |
| 2 | 系统 | 审核红冲单 | `账面库存（SKU001）：{旧} → {旧 - 原入库数}`<br>`原入库单 关联红冲单号 = 红冲单号` | 库存按负数反向调整 |
| 3 | 采购员 | 新建修正单，填入正确数量 | `修正单号 = PO20260419-0002（新建）` | 修正单与红冲单关联 |

**数字校验**：

| 时间点 | SKU001账面库存 | 说明 |
| :--- | :--- | :--- |
| 红冲前 | {N} | 原错误入库后的库存 |
| 红冲后 | {N - 原错误数量} | 账面恢复到入库前水平 |
| 修正单入库后 | {N - 原错误数量 + 正确数量} | 最终正确库存 |

**走查清单**：

- [ ] 红冲单金额 = 原单金额的负值（货币符号不变，数值取负）
- [ ] 红冲后库存变化方向：是减回去，还是直接清零？
- [ ] 原单的状态是否变更（如原单关联红冲后打标记，不允许再次红冲）？
- [ ] 红冲的权限限制：哪些角色可以发起红冲？（见主PRD权限矩阵）

**结论**：✅ / ❌

### 案例 5：边界与并发

> **目标**：验证并发操作、幂等约束、数量边界的系统行为是否符合预期。
>
