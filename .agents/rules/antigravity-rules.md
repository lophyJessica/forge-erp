# Karpathy 行为准则（AI 写代码 4 原则）

用于减少 LLM 写代码的常见通病。可与项目自身规则合并使用。
来源：Andrej Karpathy 关于 LLM 编码陷阱的观察（MIT）。

**权衡：** 本准则偏向"稳"而非"快"。对琐碎任务（改错别字、显而易见的一行改动）自行判断，不必全套照搬。

## 1. 先想再写（Think Before Coding）
**别擅自假设。别藏着疑问。把取舍摆出来。**
- 明确说出你的假设；不确定就问。
- 存在多种理解时，全部列出——不要闷头选一个。
- 若有更简单的做法，直说；该反驳时就反驳。
- 有不清楚的地方，停下来，点名哪里不清楚，然后问。

## 2. 极简优先（Simplicity First）
**用解决问题的最少代码。不写任何投机性的东西。**
- 不加没要求的功能。
- 不给一次性代码搞抽象。
- 不加没被要求的"灵活性"/"可配置性"。
- 不为不可能发生的场景写错误处理。
- 200 行能压到 50 行，就重写。
- 自问："资深工程师会不会觉得这过度复杂？"会，就简化。

## 3. 外科手术式改动（Surgical Changes）
**只碰非碰不可的地方。只清理你自己制造的烂摊子。**
- 别"顺手美化"相邻代码、注释或格式。
- 别重构没坏的东西。
- 匹配现有代码风格，哪怕你更喜欢别的写法。
- 发现无关的死代码——提示一下，别删。
- 你的改动导致某些 import/变量/函数没用了 → 删掉它们；但不要删本来就存在的死代码，除非被要求。
- 检验标准：每一行改动都能直接追溯到用户的需求。

## 4. 目标驱动执行（Goal-Driven Execution）
**定义成功标准，循环直到验证通过。**
- "加校验" → "为非法输入写测试，再让它们通过"
- "修 Bug" → "先写一个能复现 Bug 的测试，再让它通过"
- "重构 X" → "确保重构前后测试都通过"
- 多步任务先列简要计划，每步一行 `步骤 → 验证：检查点`。
- 强成功标准能让模型自主循环推进；弱标准（"能跑就行"）会逼你不停澄清。

---

**这套准则生效的标志：** diff 里没有多余改动、不再因过度复杂而返工、澄清性提问发生在动手之前而不是出错之后。
