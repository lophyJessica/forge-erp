# Forge

本项目围绕进销存系统的需求分析、PRD 输出和原型设计。
核心业务场景：批发销售、零售收银、采购入库、库存管理、往来收付款。

## 核心规范

1. **所有文档用简体中文输出**，字段名、组件名、枚举值保留英文
2. **PRD 文档必须包含**：业务背景、功能范围（In/Out Scope）、单据定位、业务场景、状态机、字段清单
3. **状态变更必须通过动作按钮触发**，不允许在表单中直接修改状态字段
4. **主数据不提供删除**，统一用"停用"操作替代

## 已知坑点

- 采购订单的"未入库数量"是计算字段，不要手填
- 采购入库时，"订单数量 ≠ 实收数量 ≠ 入库数量"——三个口径必须分开记录
- 商品名称/条码/规格在单据中为快照存储，商品档案修改后不影响已有单据

## 文档权威层级（写 PRD 前必读）

```
context/  >  templates/  >  prd-docs/
```

| 层级 | 角色 | 规则 |
| :--- | :--- | :--- |
| **context/** | 业务事实、边界、术语 | 最高权威；与 template 冲突时以 context 为准 |
| **templates/** | 章节顺序、表格格式、产出物类型 | 决定「怎么写」 |
| **prd-docs/** | 生成物 | **不是**规则源；改规则须回溯 context 或 templates |

**禁止**：只在 PRD 里改业务规则，却不更新 context/。

## PRD 套件：一套产出物，不是一份文档

### 业务单据类（如采购入库单）

| 文件 | 只管什么 | 禁止写什么 |
| :--- | :--- | :--- |
| `{单据}主PRD.md` | 背景、范围、定位、场景、规则摘要、验收 | 重复定义字段枚举/必填性 |
| `{单据}字段清单.md` | 字段唯一定义（SSOT） | 页面布局、按钮文案 |
| `{单据}_业务规则规格.md` | 状态机、校验、权限、计算规则唯一定义 | 重复写字段本体 |
| `{单据}_业务流程推演.md` | 动态沙盘：步骤 + 真实示例数据 | 与用例推演重复的细算账 |
| `{单据}_用例数据推演.md` | 带数字验证：公式、状态快照、走查清单 | UI 控件规格 |
| `{单据}_Demo_*.md` | 页面结构、控件、交互、Mock | 字段枚举、计算规则 |

### 查询页类（如库存流水、即时库存）

| 差异点 | 说明 |
| :--- | :--- |
| **无状态机** | 主 PRD 保留「状态机」章节标题，写「不适用」及原因，不要脑补草稿/审核态 |
| **无新增/编辑/详情** | 通常只有 `_Demo_列表页.md` |
| **字段清单结构** | 分「查询条件」「列表展示」「系统字段」，不用「新增页/编辑页」列 |
| **业务规则规格** | 写权限、筛选逻辑、计算/展示规则，不写状态流转表 |

### 生成时的固定输入

1. 读 `context/` 全部（至少 00、04、05、07、08 + 场景相关章节）
2. 读 `templates/` 对应模板
3. 读 **同类型参照物** PRD 套件（结构、表格、语气对齐）
4. 提示词里写清 **关键约束**（模块特有规则），改模块名可复用同一套提示词

## 查询页 vs 业务单据：AI 判断清单

| 判断 | 业务单据 | 查询页 |
| :--- | :--- | :--- |
| 用户能否创建记录 | 能（新增/保存/确认） | 不能，只读 |
| 是否有状态字段 | 有，且仅按钮驱动 | 无 |
| 库存/应付何时变 | 确认类动作触发 | 不变，只展示 |
| Demo 页数 | 列表 + 新增编辑 + 详情（+ 作业页） | 通常仅列表 |
| context/04 能力名 | 单据名（采购入库单） | 可能用能力名（库存查询），目录可用更细名（即时库存查询） |

**模块别名**：PRD 目录名与 context/04 不一致时，必须在主 PRD「能力定位」写清对应关系，并视情况回写 context/04。

## AI 走查：过关再写前端

走查标准来自 **context/ + templates/**，不是凭常识。

### 必查六维

1. **字段完整性** — 字段清单是否 SSOT？下游 Demo 是否重复定义枚举？
2. **状态机闭环**（单据类） — 有无到不了的态？执行层是否误加「待审核/已审核」？
3. **规则一致性** — 与 context/07 约束是否矛盾？
4. **术语一致性** — 对照 context/08（如用「现存」不用「账面库存」；按钮用「搜索」不用「查询」）
5. **场景覆盖** — context/06 关键异常是否进 PRD（超收阻断、关闭≠作废、整单拒收等）
6. **模块间一致** — 上下游字段来源是否存在（如即时库存依赖的「安全库存」须在商品档案定义）

### 走查流程

1. **先列问题清单**（严重程度 + 文件 + 修正方向），**不要**未确认就批量改文件
2. 按优先级修：**跨文档阻断 > context 地基 > 单模块 PRD > Demo 文案**
3. 修规则时：**字段问题只改字段清单；规则问题只改主PRD/业务规则规格；页面问题只改 Demo**

## AI 常见错误（本项目实战归纳）

### 高 — 阻断性

| 错误 | 表现 | 正确做法 |
| :--- | :--- | :--- |
| **执行层加审核流** | 采购入库单出现「待审核/已审核」 | 执行层仅「草稿→已确认」+ 可选「已作废」；审核在订单层 |
| **上下游字段不存在** | 查询页用「安全库存」，商品档案 Out Scope 且无字段 | 先确认字段在**上游字段清单**存在，或从查询页 Out Scope 拿掉 |
| **单号体系自创** | 库存流水用 `FL` 前缀但未写入 context/05 | 新单号必须进 context/05（及 context/07 前缀一览） |
| **与 context 边界冲突** | 一期 In Scope 写「自动预警推送」，context/07 标二期 | Out Scope 写清二期能力；一期只做「轻支持」并写明方式 |

### 中 — 遗漏或矛盾

| 错误 | 表现 | 正确做法 |
| :--- | :--- | :--- |
| **查询页脑补状态机** | 给流水查询设计 Tab：草稿/已确认 | 查询页无单据状态；最多按方向/类型筛选 |
| **筛选条件无效组合** | 「来源类型=全部」不算条件，用户只选类型无法搜索 | 明确哪些算「有效条件」；类型≠全部、方向≠全部可计入 |
| **模块命名漂移** | 写「即时库存查询」，context 只有「库存查询」 | 主 PRD 加别名；context/04 补映射行 |
| **关闭 vs 作废 vs 已完成** | PI 提示「PO 已完结」与 PO 关闭文案不一致 | 区分：关闭→已完成（缺量完结）、作废→已作废；文案与 PO R44 对齐 |
| **三口径缺一项** | 即时库存只展示「现存」 | 现存、占用、可用同屏（context/07 QTY-06） |

### 低 — 一致性与体验

| 错误 | 表现 | 正确做法 |
| :--- | :--- | :--- |
| **术语混用** | 「账面库存」「可用库存」「预占」 | 统一：现存、占用、可用（context/08） |
| **字段清单与 Demo 列不一致** | 清单写列表展示「流水号」，Demo 无该列 | 改 Demo 或改清单，导出列同步 |
| **Demo 文件命名不统一** | `_Demo_PRD.md` vs `_Demo_列表页.md` | 查询/列表页统一 `{模块}_Demo_列表页.md` |
| **Mock 日期过时** | 仍用 2024、2025 | 与演示环境一致（如 2026） |
| **参照物套件不完整** | 用只有 5 份的 PO 对比 8 份的 PI | 走查时注明参照范围；或补全参照模块套件 |

## 可复用的生产力公式

```
地基（context/）
  + 模板（templates/）
  + 参照物 PRD 套件（同类型、最好已走查）
  + 模块提示词（改名称 + 关键约束）
  + 并行 Agent（独立模块各跑一套）
  → PRD 套件初稿
  → 走查清单（不过关不写前端）
  → 按清单回溯修改（先 context，再 prd-docs）
```

### 提示词里建议固定的「关键约束」块

- 页面类型：单据 / 查询（决定是否要状态机）
- 数量口径：如三数量分离、库存三口径公式
- 单据来源：如 PI 必须来自 PO、流水必须关联来源单号
- 一期 Out Scope：直接引用 context/07，避免 AI 扩 scope

### 并行生成注意

- 两 Agent **共享同一套** context + templates + 参照物
- 只改：**模块名、输出目录、关键约束**
- 生成后做 **跨模块走查**（流水现存与即时库存公式、字段来源是否一致）
