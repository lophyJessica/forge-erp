关键业务约束（来自 context/）：
- 采购入库单属于"第2层——业务执行层"，必须基于已审核的采购订单创建
- 一张采购订单可对应多张采购入库单（1:N），支持分批入库
- 入库数量 ≤ 实收数量 ≤ 采购订单的未入库数量
- 入库确认后：库存增加 + 采购订单累计入库数量回写 + 应付形成
- 入库单是执行层单据，不需要审核状态——核心动作只有"确认入库"
- 状态变更通过动作按钮触发，不在表单中直接修改状态字段
- 术语体系以 context/08-术语与口径字典.md 为准
- 所有文档风格参照采购订单 PRD 套件，保持结构、术语、表格格式一致

## 讲师备注

**预期产出**：
- 6 类文件（主PRD + 字段清单 + 业务规则规格 + 流程推演 + 用例推演 + 3 Demo PRD），PRD 套件初稿级别
- 状态机体现代执行层单据特点——状态简洁，不出现审核类状态
- 字段清单正确区分三数量口径

**点评要点**：
- 先展示采购订单的完整 PRD 套件做参照物——让学员看到"成品"长什么样
- 如果 AI 在状态机里加了"已审核"状态——指出执行层单据不需要审核，AI 凭通用经验脑补了
- 如果 AI 漏了用例数据推演中的"分批到货"场景——提示词里只给了单次推演的例子，AI 不会主动扩展
- 强调：PRD 不是"一个文档"，是一套分层产出物。每份文件各管各的，互不越界——这个原则后续精讲课展开
