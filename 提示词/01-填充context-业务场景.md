```
这个文件已经有四个场景的骨架，请按以下要求逐一填充：

场景一「采购下单与到货入库」——本次演示重点，需要最详细：
- 参与角色表格（采购员、仓管员、供应商、财务）
- 主流程 Mermaid flowchart：从"创建采购订单"到"入库完成+应付形成"
- 必须覆盖以下异常分支：
  * 部分到货（本次只到了一部分，剩余待补货）
  * 全部拒收（质量不合格，整单退回）
  * 采购退货（已入库后发现需要退，触发退货流程）
  * 缺量完结（供应商无法补货，人工关闭订单）
- 关键业务规则：至少 5 条（参考 context/05 的规则体系），用表格输出

场景二「渠道批发接单与发货」：
- 参与角色 + 主流程 Mermaid（正常+库存不足+信用超额预警）
- 关键业务规则至少 3 条

场景三「零售收银」：
- 参与角色 + 主流程 Mermaid（正常+折让超权限需主管授权）
- 关键业务规则至少 3 条

场景四「客户收款与应收处理」：
- 参与角色 + 主流程 Mermaid（正常+不明款暂挂+部分核销）
- 关键业务规则至少 3 条

约束：
- 状态变更遵循 context/05 的规则（通过动作按钮触发，不在表单中直接修改）
- 使用 context/05 中定义的术语和单号前缀
- 采购订单与入库单的关系是 1:N——一张订单可多次入库
- 强盛一期没有复杂审批流，采购订单为管理员一级审核
```

## 讲师备注

**预期产出**：
- 四个场景的完整 Mermaid 流程图，采购场景异常分支最丰富
- 每条业务规则有明确的约束表述

**点评要点**：
- 如果 AI 把"采购订单→采购入库"画成 1:1 关系，纠正为 1:N——这是进销存的经典设计
- 指出 AI 正确遵循了 context/05 的状态变更规则——这就是为什么要把规则写进 context
- 如果 AI 在流程图中加了"审批节点"，确认是否在 context 范围内——强盛一期没有复杂审批流
- 这一步的顺序很关键：context/06 必须在 context/04 之前填充——先有场景再有模块
