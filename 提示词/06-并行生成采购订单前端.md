## 提示词（总控——先建工程，再并行分发页面）

```
请先完成以下两步：

第一步：工程初始化
在 front-prototype/ 目录下创建最小 React 工程：
- Vite + React + TypeScript + Tailwind CSS + shadcn/ui
- 创建 src/types/ 目录存放共享类型定义（从采购订单字段清单派生）
- 创建 src/api/ 目录存放共享 API 接口层
- 安装依赖并确保 npm run dev 可启动

第二步：并行生成 5 个页面

页面必须共享同一份类型定义（src/types/purchaseOrder.ts）和 API 层（src/api/purchaseOrder.ts）。

═══════════════════════════════════════
页面 1：采购订单列表页
路径：front-prototype/src/pages/PurchaseOrderList.tsx

基于：采购订单_Demo_列表页.md
- 顶部：页面标题 + "新增采购订单"按钮
- 筛选区：采购单号（Input）、供应商（SelectWithSearch）、入库仓库（SelectWithSearch）、订单状态（Select：全部/待审核/待入库/部分入库/已完成/已作废）、下单日期范围（DateRange）
- 表格列：采购单号、供应商、入库仓库、下单日期、预计到货日期、订单状态（Badge）、采购总金额、最后修改时间
- 操作列：查看详情、编辑（仅草稿态可用）、作废（待审核/待入库态可用）
- 底部分页
- 每行支持勾选（Checkbox），表格顶部提供"批量导出"按钮

页面 2：采购订单新增编辑页
路径：front-prototype/src/pages/PurchaseOrderForm.tsx

基于：采购订单_Demo_新增编辑页.md
- 通过路由参数区分新增/编辑模式
- 头部信息区：供应商（必填）、入库仓库（必填）、下单日期（默认当天）、预计到货日期（选填）、采购备注（选填）
- 商品明细表格：可增删行，每行包含：商品选择器（搜索编码/名称）、商品名称（自动带出只读）、规格型号（自动带出只读）、单位（自动带出只读）、采购数量（必填）、单价含税（必填）、税率（下拉选）、金额（自动计算只读）、行备注（选填）、删除按钮
- 底部汇总：商品种数、采购总数量、采购总金额
- 底部操作：保存草稿、提交审核
- 编辑模式：草稿态加载已有数据，可编辑；非草稿态加载已有数据，关键字段锁定只读

页面 3：采购订单详情页
路径：front-prototype/src/pages/PurchaseOrderDetail.tsx

基于：采购订单_Demo_详情页.md
- 状态区：当前状态 Badge + 状态时间线
- 基础信息区：供应商、入库仓库、下单日期等（只读）
- 商品明细表格：含"累计已入库数量"和"未入库数量"列
- 底部操作按钮根据状态动态变化：
  待审核 → "审核通过""驳回"
  待入库/部分入库 → "创建入库单"
  已完成 → 无操作按钮
  已作废 → 显示作废原因
- 关联入库单列表（如有）

页面 4：采购订单导入
路径：front-prototype/src/pages/PurchaseOrderImport.tsx

- 文件上传区：拖拽或点击上传，支持 .xlsx/.csv
- 上传后预览：表格展示解析后的数据，标出校验失败的行（红色高亮）
- 字段映射：如果文件列名与系统字段不一致，提供下拉选择映射
- 校验规则：供应商必须存在于系统、仓库必须存在、商品编码必须有效、数量必须 > 0
- 底部：确认导入（全部校验通过后可用）+ 取消
- 导入完成后显示结果摘要：成功 N 条、失败 M 条，失败行可下载错误报告

页面 5：采购订单导出
路径：front-prototype/src/pages/PurchaseOrderExport.tsx

- 导出范围：当前筛选结果 / 全部订单 / 选中订单
- 导出字段选择：Checkbox 列表，默认全选，支持"全选/取消全选"
- 导出格式：Excel(.xlsx) / CSV(.csv)
- 导出按钮 + 取消按钮
- 导出中显示进度，完成后自动下载

═══════════════════════════════════════

关键约束（所有页面通用）：
- 页面风格：B 端管理后台——信息密度高、操作直接，不是展示站或官网风格
- 状态 Badge 颜色规范：草稿=灰色、待审核=蓝色、待入库=黄色、部分入库=橙色、已完成=绿色、已作废=红色
- 金额字段右对齐，保留2位小数
- 日期格式统一为 YYYY-MM-DD
- 所有状态变更通过按钮触发弹窗确认，不在表单中直接修改状态字段
- 审核后关键字段锁定为只读（供应商、仓库、商品明细、单价）
- 所有页面共享 src/types/purchaseOrder.ts 和 src/api/purchaseOrder.ts
```

## 讲师备注

**预期产出**：
- 一个最小前端工程 + 5 个页面骨架（列表/表单/详情/导入/导出）
- 共享类型定义和 API 接口层
- 列表页的筛选区、表格列、状态 Badge 颜色大面上正确
- 表单页的商品选择器、自动计算金额逻辑基本跑通
- 详情页的状态时间线和条件按钮位置正确
- 导入导出页面骨架完整

**点评要点**：
- 先看风格：是不是 B 端管理后台的密度（紧凑、信息量大），而不是展示站风格
- 再看数据：字段名、状态枚举值是否正确对齐字段清单
- 五个页面同时出来——强调多 Agent 并行效率
- 如果 AI 在表单页放了"状态"下拉——这是违规的，状态通过动作按钮变更
- 导入导出是 B 端高频需求——展示 AI 能一键出骨架，不需要从零写
