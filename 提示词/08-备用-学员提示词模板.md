# 08-备用：学员课后提示词模板

> 以下是给学员课后自己练习用的最小提示词集。复制即用，改项目名和模块名即可。
> 对齐第 4 节演示路线：context 填充（先场景后模块）→ PRD 套件（6 类产出物）→ 并行 Agent → 前端生成

## A. 验证地基

```
请读取我项目的 context/ 下所有文件和 AGENTS.md。
基于这些内容，用一段话概括这个项目的业务范围，然后列出核心模块清单。
```

## B. 填充 context（按第 4 节演示顺序）

```
请读取我项目的 context/ 下现有文件。
请按以下顺序帮我填充 context/ 的缺口文件：

第一步：填充核心业务场景（context/06）
基于已有的业务背景，为每个核心场景输出：参与角色、Mermaid 流程图（含正常路径和异常分支）、关键业务规则。

第二步：从场景反推功能模块清单（context/04）
基于第一步填好的业务场景，反推系统需要哪些功能模块来支撑这些流转。列出模块树、模块间数据依赖图、一期/二期分界。

第三步：汇总业务约束（context/07）
读取全部 context/ 文件，提取所有约束规则，按类别整理成"宪法级"约束文档。每条约束标注来源文件。
```

## C. 生成 PRD 套件（6 类产出物）

```
请基于 context/ 全部文件和 templates/ 下的规范，为 [模块名称] 生成完整的 PRD 套件初稿。

这不是一份文档，是一套分层产出物：

1. [模块名称]主PRD.md —— 需求背景、功能范围、单据定位、业务场景、状态机、业务规则、验收标准
2. [模块名称]字段清单.md —— 所有字段的完整定义（字段名、类型、必填、默认值、数据来源）
3. [模块名称]_业务规则规格.md —— 状态机图、状态流转动作矩阵、校验规则、权限规则
4. [模块名称]_业务流程推演.md —— 正常路径 + 关键异常分支 Mermaid 流程图
5. [模块名称]_用例数据推演.md —— 带具体数据跑一遍，验证字段、状态、计算是否自洽
6. 前端 Demo PRD —— 列表页、操作页、详情页各一份

每份文件有且只有它该管的信息，互不越界。
参照 [已有的完整模块] 的 PRD 套件——保持文档结构、表格格式、描述风格一致。
```

## D. 并行生成多个模块 PRD

```
我有两个模块需要同时生成 PRD 套件：[模块A] 和 [模块B]。

请用两个独立的 Agent 并行处理：
- Agent 1：为 [模块A] 生成完整 PRD 套件（6 类产出物），参考 [参照物模块] 的风格
- Agent 2：为 [模块B] 生成完整 PRD 套件（6 类产出物），参考 [参照物模块] 的风格

两个模块共享同一套 context/ 和 templates/，但输出到各自的 prd-docs/ 子目录。
```

## E. 并行生成前端页面

```
请基于 [模块名称] 的完整 PRD 套件（主PRD + 字段清单 + Demo PRD），生成前端页面。

技术栈：React + TypeScript + Tailwind CSS + shadcn/ui
页面风格：B 端管理后台，信息密度高，操作直接。

请用多个独立的 Agent 并行生成以下页面：
- Agent 1：列表页（筛选区 + 表格 + 分页 + 批量操作）
- Agent 2：新增编辑页（表单布局 + 字段校验 + 提交逻辑）
- Agent 3：详情页（信息展示 + 状态时间线 + 条件按钮）
- Agent 4：导入功能（文件上传 + 字段映射 + 校验 + 结果摘要）
- Agent 5：导出功能（范围选择 + 字段选择 + 格式选择）

所有页面共享同一份类型定义和 API 接口层。
所有状态变更通过按钮触发弹窗，不在表单中直接修改状态字段。
```

## F. 走查自检

```
请重新读取刚才生成的所有 PRD 套件和前端代码。

对照 context/ 业务规则和 templates/ 规范，逐项检查：
1. 字段是否遗漏？
2. 状态流转是否闭环？
3. 校验规则是否有矛盾？
4. 页面交互是否违规？（如表单中直接放了状态修改字段）
5. 术语是否一致？

列出所有问题，标注严重程度（高/中/低）和修正方向。
不直接修改文件，等我确认后再逐条修。
```
