# 03-生成 context/07：本期项目边界与业务约束

> 演示步骤：第 3 步（context 填充的最后一步）
> 位置：context/06 和 04 都填完后，汇总所有约束

## 喂入文件

@context/（全部文件：00-06）
@AGENTS.md

## 提示词（直接复制）

```
请读取 context/ 下全部文件（00-06）和 AGENTS.md，然后帮我填写 context/07-本期项目边界与业务约束.md。

这个文件是后续写所有 PRD 的"宪法级"输入——每次生成 PRD 时 @ 进去，AI 会自动遵守这些约束。

要求：

一、In Scope（本期必做）
用表格列出所有本期要实现的功能点，按模块分组，标注优先级（P0/P1/P1.5）。
对齐 context/04 的一期模块清单和 context/06 的业务场景覆盖范围。

二、Out of Scope（本期明确不做）
用表格列出，每条附上"不做原因"和"后续计划"。至少包含：
- 自动补货建议
- 安全库存预警
- 信用额度自动硬拦截
- 复杂审批流
- 复杂经营分析报表
- 电商代发扩展

三、本期轻支持（做但不做重）
用表格列出，每条附上"轻支持方式"和"原因"。至少包含：
- 采购价格参考与改价留痕
- 少收/部分到货/拒收处理
- 基础应付查询

四、关键业务约束汇总
汇总 context/ 中所有文件里出现的约束规则，按类别整理：
- 主数据约束（如不可删除、停用代替）
- 状态变更约束（如必须通过动作按钮触发）
- 单据约束（如审核后关键字段锁定、商品信息快照存储）
- 数量口径约束（如采购数量≠实收数量≠入库数量，三口径独立记录）
- 权限与操作约束（如术语字典中的动词规范）

五、跨模块通用规则
- 单号规则（前缀+日期+流水号，见 context/05）
- 权限模型摘要
- 操作留痕要求

请直接输出完整文件内容。每一条约束标注来源（出自 context/ 哪个文件的哪个章节）。
```

## 讲师备注

**预期产出**：
- 一份结构化的约束汇总文件，相当于 context/ 全部内容的"宪法摘要"
- In/Out Scope 与 context/02 一期判断 + context/04 模块清单对齐
- 业务约束均有来源标注

**点评要点**：
- 这个文件是后续写 PRD 时最重要的输入——@context/07 进去，AI 自动遵守约束
- 如果 AI 漏了某条约束，说明 context/ 里该约束表述不够明确——地基质量的反馈循环
- 这一步完成后，context/ 全部就位——地基完整，可以开始产出链路
